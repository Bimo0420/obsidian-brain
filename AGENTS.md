как# AGENTS.md - Схема персональной базы знаний

> Адаптировано на основе архитектуры [LLM Knowledge Base Андрея Карпати](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).
> Вместо поглощения внешних статей эта система компилирует знания из ваших собственных диалогов с ИИ.

## Аналогия с компилятором

```
daily/          = исходный код   (ваши диалоги — сырой материал)
LLM             = компилятор     (извлекает и организует знания)
knowledge/      = исполняемый файл (структурированная база знаний)
lint            = тесты          (проверки здоровья на согласованность)
queries         = рантайм        (использование знаний)
```

Вы не организуете знания вручную. Вы ведете диалоги, а LLM берет на себя синтез, перекрестные ссылки и обслуживание.

---

## Архитектура

### Уровень 1: `daily/` — Логи диалогов (неизменяемый источник)

Ежедневные логи фиксируют то, что происходило в ваших сессиях кодинга с ИИ. Это "сырые источники" — они только дополняются и никогда не редактируются после создания.

```
daily/
├── 2026-04-01.md
├── 2026-04-02.md
├── ...
```

Каждый файл следует такому формату:

```markdown
# Daily Log: YYYY-MM-DD

## Sessions

### Session (HH:MM) - Краткий заголовок

**Контекст:** Над чем работал пользователь.

**Ключевые моменты:**
- Пользователь спросил про X, ассистент объяснил Y
- Решили использовать подход Z, потому что...
- Обнаружили, что W не работает, когда...

**Принятые решения:**
- Выбрали библиотеку X вместо Y, так как...
- Архитектура: остановились на паттерне Z

**Извлеченные уроки:**
- Всегда делайте X перед Y, чтобы избежать...
- Особенность Z заключается в том, что...

**Задачи:**
- [ ] Разобраться с X
- [ ] Рефакторинг Y, когда будет время
```

### Уровень 2: `knowledge/` — Скомпилированные знания (принадлежат LLM)

LLM полностью владеет этой директорией. Люди читают её, но редко редактируют напрямую.

```
knowledge/
├── index.md              # Главный каталог — каждая статья с резюме в одну строку
├── log.md                # Лог сборки (только дозапись)
├── concepts/             # Атомарные статьи о концепциях
├── connections/          # Связи между 2+ концепциями
└── qa/                   # Сохраненные ответы на запросы
```

### Уровень 3: `raw/` и `processed/` — Внешние источники знаний

Помимо диалогов, вы можете загружать в базу внешние материалы (статьи, PDF, код, репозитории).

```
raw/            # Входящие материалы для обработки
processed/      # Архив уже обработанных материалов
```

Вы кладете файлы или папки в `raw/`, запускаете скрипт `process_raw.py`, и LLM самостоятельно изучает их, извлекает концепции в `knowledge/`, а затем перемещает исходник в `processed/`.

### Уровень 4: Этот файл (AGENTS.md)

Схема, которая говорит LLM, как компилировать и поддерживать базу знаний. Это "спецификация компилятора".

---

## Структурные файлы

### `knowledge/index.md` — Главный каталог

Таблица со списком всех статей. Это основной механизм извлечения — LLM читает этот файл ПЕРВЫМ при ответе на любой запрос, а затем выбирает нужные статьи для полного прочтения.

Формат:

```markdown
# Knowledge Base Index

| Article | Summary | Compiled From | Updated |
|---------|---------|---------------|---------|
| [[concepts/supabase-auth]] | Паттерны RLS и нюансы JWT | daily/2026-04-02.md | 2026-04-02 |
| [[connections/auth-and-webhooks]] | Паттерны проверки токенов в Supabase и Stripe | daily/2026-04-02.md, daily/2026-04-04.md | 2026-04-04 |
```

## Форматы статей

### Статьи о концепциях (`knowledge/concepts/категория/`)

Одна статья на одну атомарную единицу знаний: факты, паттерны, решения, предпочтения.
**Важно:** Статьи должны быть сгруппированы по логическим подпапкам (например, `concepts/ml/`, `concepts/devops/`, `concepts/web/`), а не лежать в корне `concepts/`.

```markdown
---
title: "Название концепции"
aliases: [альтернативное-название]
tags: [домен, топик]
sources:
  - "daily/2026-04-01.md"
created: 2026-04-01
updated: 2026-04-01
---

# Название концепции

[Core-объяснение в 2-4 предложениях]

## Ключевые моменты

- [Пункты, каждый из которых самодостаточен]

## Детали

[Глубокое объяснение в стиле энциклопедии]

## Связанные концепции

- [[concepts/related-concept]] - Описание связи

## Источники

- [[daily/2026-04-01.md]] - Первичное обнаружение при настройке проекта
```

## Основные операции

### 1. Компиляция (daily/ -> knowledge/)

При обработке ежедневного лога:
1. Читает файл лога.
2. Читает `knowledge/index.md`, чтобы понять текущее состояние базы.
3. Обновляет существующие статьи или создает новые концепции (`concepts/`).
4. Создает статьи о связях (`connections/`), если обнаружена неочевидная зависимость.
5. Обновляет индекс и лог сборки.

### 2. Запрос (Query)

1. Читает `knowledge/index.md`.
2. Идентифицирует 3-10 релевантных статей.
3. Читает их полностью.
4. Синтезирует ответ с цитатами в виде `[[вики-ссылок]]`.

### 3. Линтинг (Lint)

Проверки: битые ссылки, сиротские страницы, нескомпилированные логи, устаревшие статьи, противоречия.

## Система хуков (Автоматический захват)

Хуки настраиваются в `.gemini/settings.json` и срабатывают автоматически.

- **`session-start.py`**: Внедряет индекс базы знаний в начало каждой сессии.
- **`session-end.py`**: Захватывает транскрипт и запускает фоновую очистку (`flush.py`).
- **`pre-compact.py`**: Страховка — захватывает контекст перед тем, как CLI его сожмет (суммирует).

### Фоновый процесс очистки (`flush.py`)

1. Читает контекст из временного файла.
2. Вызывает LLM для анализа: что из этого диалога стоит сохранить?
3. Добавляет результат в `daily/YYYY-MM-DD.md`.
4. Если время > 18:00, запускает `compile.py`.

## Зависимости и инструменты

- **Python 3.12+**, управление через **uv**.
- **Gemini CLI** (используется для безголовых запросов в обновленной версии).
- Данные хранятся в чистом Markdown, что позволяет использовать **Obsidian** для визуализации графа связей.
