rj> Девять ключевых дисциплин, которые превращают эксперименты по машинному обучению в надёжные промышленные системы.

**MLOps (Machine Learning Operations)** - это набор практик и инструментов, которые превращают ML-модели из «лабораторных экспериментов» в надёжно работающие промышленные системы.
**MLOps-инженер — это операционный директор ресторана.** Он не придумывает рецепты, но строит кухонную инфраструктуру, конвейеры, контроль качества и систему поставок, чтобы всё работало стабильно.
## Ключевые компоненты
MLOps охватывает полный жизненный цикл ML-системы:[](https://blog.skillfactory.ru/chto-takoe-mlops/)​
- **Управление данными** — версионирование датасетов, контроль качества, data pipelines
- **Разработка и обучение моделей** — эксперименты, трекинг метрик, воспроизводимость
- **Версионирование моделей** — хранение артефактов, Model Registry
- **Тестирование и валидация** — тесты данных, кода, модели и инфраструктуры
- **Деплой в продакшен** — CI/CD для моделей, A/B тесты, canary-деплой
- **Мониторинг** — отслеживание качества модели, data drift, бизнес-метрики

## Уровни зрелости (Maturity Levels)
В MLOps выделяют три уровня автоматизации:[](https://mws.ru/blog/chto-takoe-ml-ops/)​
1. **Level 0** — ручные процессы: модели деплоятся вручную, нет CI/CD, минимум автоматизации
2. **Level 1** — автоматизация ML-пайплайна: непрерывное обучение, автоматический триггер переобучения
3. **Level 2** — полный CI/CD: автоматизированное тестирование кода, данных и инфраструктуры; сквозное версионирование от исходных данных до артефактов модели

## MLOps vs DevOps
MLOps наследует принципы DevOps (CI/CD, IaC, мониторинг), но имеет важную специфику:[](https://bigdataschool.ru/wiki/mlops/)

| Аспект          | DevOps                 | MLOps                                   |
| --------------- | ---------------------- | --------------------------------------- |
| Артефакт        | Код/приложение         | Модель + код + данные                   |
| Версионирование | Git для кода           | Git + DVC для данных и моделей          |
| Тестирование    | Unit/integration тесты | + тесты на data drift, model decay      |
| Триггер CI/CD   | Изменение кода         | + изменение данных, деградация метрик   |
| Мониторинг      | Uptime, latency        | + качество предсказаний, бизнес-метрики |
## Ключевые инструменты
- **Эксперименты и трекинг:** MLflow, Weights & Biases, Langfuse (для LLM)
- **Оркестрация пайплайнов:** Apache Airflow, Prefect, n8n
- **Версионирование данных:** DVC, Delta Lake
- **Деплой и сервинг:** BentoML, Triton Inference Server, Ollama
- **Мониторинг:** Prometheus + Grafana, Evidently AI
- **Контейнеризация:** Docker, Kubernetes

## MLOps для LLM (LLMOps)
С распространением больших языковых моделей появился подвид — **LLMOps**. Его специфика: управление промптами и их версиями, оценка качества генерации (RAGAS, LLM-as-a-judge), мониторинг галлюцинаций, оптимизация inference (квантизация, batching), а также работа с RAG-пайплайнами.

## 1. Version Control
**Определение:** система управления версиями отслеживает изменения в коде, данных и конфигурациях, позволяя команде работать параллельно и откатываться к любому состоянию проекта.

В MLOps под контролем версий находится не только код, но и **датасеты, веса моделей, гиперпараметры и конфигурации пайплайнов**. Это основа воспроизводимости экспериментов.

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|Git + GitHub/GitLab|Код обучения, preprocessing-скрипты, Notebooks|
|DVC (Data Version Control)|Версионирование больших датасетов и артефактов модели|
|MLflow / W&B|Версионирование экспериментов, метрик и параметров|

### Связи с другими дисциплинами

- **→ CI/CD:** каждый git-push триггерит пайплайн сборки и тестирования
- **→ IaC:** конфиги инфраструктуры хранятся в том же репозитории
- **→ Monitoring:** тег версии модели позволяет сопоставить деградацию с конкретным коммитом

## 2. CI/CD
**Определение:** Continuous Integration / Continuous Delivery — практика автоматического построения, тестирования и доставки кода или моделей в среду эксплуатации после каждого изменения.

### Этапы пайплайна
Код проходит последовательные стадии:[](https://yandex.cloud/ru/blog/ci-cd-pipeline)​

1. **Push** — разработчик отправляет изменения в Git, это запускает весь процесс
2. **Build** — сборка, установка зависимостей, статический анализ, линтеры; для Docker-проектов — сборка образа
3. **Unit-тесты** — быстрые проверки изолированных компонентов
4. **Deploy на staging** — окружение, максимально приближённое к продакшну
5. **Интеграционные / E2E тесты** — более долгие проверки на staging
6. **Deploy в продакшн** — канареечный релиз или blue-green деплоймент
7. **Мониторинг** — отслеживание метрик, алерты при аномалиях
8. **Rollback** — автоматический откат при критических проблемах[](https://yandex.cloud/ru/blog/ci-cd-pipeline)​

### Ключевые принципы
- **Автоматизировать всё возможное** — ручные операции источник ошибок; охватывать нужно не только тесты, но и конфиги, генерацию документации[](https://yandex.cloud/ru/blog/ci-cd-pipeline)​
- **Версионировать все артефакты** — код, конфиги, схемы БД, скрипты деплоя должны быть в Git
- **Быстрая обратная связь** — сначала запускать лёгкие и быстрые проверки (линтеры, юниты), затем тяжёлые (интеграционные тесты)[](https://yandex.cloud/ru/blog/ci-cd-pipeline)​
- **Идентичные процессы для всех окружений** — staging и прод отличаются только масштабом и данными, но не самим пайплайном[](https://yandex.cloud/ru/blog/ci-cd-pipeline)​
В ML-контексте CI/CD расширяется до **CT (Continuous Training)** — автоматического переобучения модели при появлении новых данных или деградации метрик.

### Пайплайн
```
git push → Lint/Tests → Build Image → Train Model → Evaluate → Deploy
```

### Примеры инструментов

| Инструмент         | Назначение                                   |
| ------------------ | -------------------------------------------- |
| GitHub Actions     | Пайплайн прямо в репозитории, YAML-конфиги   |
| GitLab CI          | Self-hosted runners, интеграция с Kubernetes |
| Jenkins / CircleCI | Гибкая оркестрация сложных пайплайнов        |
| ArgoCD             | GitOps-деплой для Kubernetes                 |
|                    |                                              |

### Связи с другими дисциплинами
- **→ Version Control:** изменение кода или данных запускает пайплайн
- **→ Containerization:** образы собираются в пайплайне и пушатся в Container Registry
- **→ Monitoring:** алерт о деградации триггерит автоматическое переобучение (CT)

## 3. ML Fundamentals
**Определение:** базовые концепции машинного обучения — типы обучения, функции потерь, метрики качества, борьба с переобучением — необходимы каждому MLOps-инженеру для осмысленной автоматизации ML-систем.

MLOps-инженер не обязан разрабатывать архитектуры нейросетей, но должен понимать **жизненный цикл модели**, чтобы правильно строить инфраструктуру вокруг неё.

### Ключевые концепции
- Supervised / Unsupervised / Reinforcement Learning
- Train / Validation / Test split — предотвращение data leakage
- Метрики: Accuracy, F1, AUC-ROC, RMSE
- Feature Engineering и Feature Store
- Hyperparameter Tuning (Optuna, Ray Tune)

### Связи с другими дисциплинами
- **→ Monitoring:** знание метрик позволяет настроить осмысленные алерты при деградации
- **→ Explainable AI:** понимание модели — предпосылка для её объяснимости

## 4. Containerization
**Определение:** упаковка приложения вместе со всеми зависимостями в изолированный контейнер, который запускается одинаково на любой машине — от ноутбука разработчика до облачного кластера.

Контейнеры решают классическую проблему MLOps: **«у меня работает, в продакшене нет»**. Модель и всё окружение (Python, CUDA, библиотеки) упакованы в один образ.

### Примеры инструментов

| Инструмент        | Назначение                                                 |
| ----------------- | ---------------------------------------------------------- |
| Docker            | Стандарт отрасли для создания образов и локального запуска |
| Kubernetes        | Оркестрация контейнеров в кластере, масштабирование        |
| BuildKit / Podman | Современные альтернативы Docker для CI-окружений           |

### Связи с другими дисциплинами

- **→ CI/CD:** образы собираются в пайплайне и пушатся в Container Registry
- **→ Orchestration:** Kubernetes запускает именно контейнеры при деплое модели

## 5. Data Engineering Fundamentals
**Определение:** проектирование и поддержка пайплайнов для сбора, трансформации, хранения и доставки данных — сырья для обучения и инференса ML-моделей.

Качество модели напрямую определяется качеством данных. Data Engineering строит **надёжный конвейер** от источника до Feature Store.

### Пайплайн данных
```
Источники → Ingestion → Transform (ETL) → Data Lake → Feature Store
```

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|Apache Spark|Распределённая обработка больших объёмов данных|
|Kafka / Flink|Стриминг данных в реальном времени|
|Feast / Tecton|Feature Store — централизованное хранилище признаков|

### Связи с другими дисциплинами
- **→ Version Control:** датасеты версионируются через DVC наравне с кодом
- **→ Monitoring:** мониторинг дрейфа данных начинается с понимания их структуры

## 6. Infrastructure as Code
**Определение:** управление серверами, сетями и облачными ресурсами через декларативный код вместо ручных кликов в консоли — инфраструктура описывается файлами и хранится в git.

IaC обеспечивает **воспроизводимость среды**: dev, staging и production создаются из одних и тех же шаблонов, что исключает «снежинки» — уникальные серверы с неизвестной конфигурацией.

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|Terraform|Облако-независимый декларативный IaC (AWS, GCP, Azure)|
|Helm Charts|Шаблонизация Kubernetes-манифестов|
|Ansible / Pulumi|Конфигурирование машин и императивный IaC|

### Связи с другими дисциплинами

- **→ Version Control:** инфра-код хранится в git наравне с кодом модели
- **→ CI/CD:** `terraform apply` запускается в пайплайне при изменении инфра-кода
- **→ Containerization:** IaC поднимает Kubernetes-кластер, в котором живут контейнеры

---

## 7. Orchestration / Deploy
**Определение:** координация последовательности задач ML-пайплайна (предобработка → обучение → оценка → деплой) и управление жизненным циклом модели в продакшене.

Оркестрация отвечает за **«когда и как»** запускать пайплайн, обрабатывать зависимости между шагами, повторять упавшие таски и масштабировать сервисы инференса.

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|Apache Airflow|DAG-оркестрация, расписание, мониторинг задач|
|Kubeflow Pipelines|ML-пайплайны нативно в Kubernetes|
|Seldon / BentoML|Деплой и серверинг ML-моделей (REST, gRPC)|
|Blue/Green Deploy|Постепенный переход трафика на новую версию модели|

### Связи с другими дисциплинами

- **→ Containerization:** оркестратор запускает контейнеры с моделью
- **→ CI/CD:** деплой — финальный шаг CD-пайплайна
- **→ Monitoring:** после деплоя начинается непрерывный мониторинг поведения модели

---

## 8. Monitoring / Observability
**Определение:** непрерывное отслеживание поведения моделей и инфраструктуры в продакшене — обнаружение деградации, data drift и системных аномалий в реальном времени.

Модели устаревают — мир меняется, а данные дрейфуют. **Без мониторинга** деградация качества может быть незаметна месяцами. MLOps-мониторинг делится на два уровня:

- **Системный:** CPU/GPU, память, латентность, ошибки (Prometheus + Grafana)
- **ML-метрики:** Data Drift, Concept Drift, распределение предсказаний (Evidently AI, Arize)

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|Grafana + Prometheus|Дашборды системных метрик и алерты|
|Evidently AI|Отчёты о дрейфе данных и качестве модели|
|Jaeger / Tempo|Трассировка запросов через микросервисы|

### Связи с другими дисциплинами

- **→ CI/CD:** алерт о деградации триггерит автоматическое переобучение (CT)
- **→ Explainable AI:** объяснения помогают понять причину деградации при дрейфе

---

## 9. Explainable AI
**Определение:** набор методов и инструментов, которые делают решения ML-моделей понятными для людей — объясняют, почему модель дала тот или иной результат.

XAI критичен в regulated domains — кредитование, медицина, юстиция — где по закону (EU AI Act, GDPR) модель **обязана объяснить своё решение**. Также помогает отлаживать баги и предубеждения.

### Примеры инструментов

|Инструмент|Назначение|
|---|---|
|SHAP|Вклад каждого признака в конкретное предсказание|
|LIME|Локальные линейные аппроксимации чёрного ящика|
|Grad-CAM|Визуализация важных зон на изображениях (CNN)|
|Fairness Indicators|Выявление предубеждений по защищённым группам|

### Связи с другими дисциплинами

- **→ Monitoring:** объяснения при деградации раскрывают, какие признаки «поплыли»
- **→ ML Fundamentals:** глубокое понимание модели — основа правильной интерпретации SHAP-значений

## Как всё связано вместе
MLOps — это не набор независимых инструментов, а **замкнутый цикл**:

```
Data Engineering
      ↓
Version Control  ←──────────────────────────────┐
      ↓                                          │
    CI/CD                                        │
      ↓                                    Monitoring
Containerization                                 │
      ↓                                    Explainable AI
     IaC                                         │
      ↓                                          │
Orchestration / Deploy ──────────────────────────┘
```

Данные обрабатываются → эксперименты версионируются → пайплайн автоматизируется → всё упаковывается → разворачивается через управляемую инфраструктуру → непрерывно отслеживается и объясняется — что снова ведёт к улучшению данных и моделей.
